# 最终需求交付模板

## 目录

1. 使用规则
2. 固定开头
3. 完整交付结构
4. 主动停止时的调整
5. 输出前一致性检查

## 1. 使用规则

根据需求类型保留适用章节。对明显不适用的章节写一句“不适用：原因”，不要生成虚假内容。对简单需求可以合并相邻章节，但不得省略关键规则、异常、验收和原型。

最终文档应让产品、设计、研发、测试、运营或业务负责人能够理解：为什么做、为谁做、做什么、如何工作、失败怎么办、如何验收。

所有结论必须标记其证据状态：

- 已确认：用户明确决定或提供可靠事实。
- 已接受假设：用户明确同意采用建议方案。
- 暂定假设：为补足方案而提出，但用户未确认。
- 非阻塞待确认：不影响整体方案，可在详细设计阶段处理。
- 关键阻塞：会显著改变方案，且仅在用户主动停止时允许保留。

## 2. 固定开头

每份最终输出都从以下区块开始：

```text
【需求澄清结果】
信息完整度：xx%
方案把握度：xx%
目标阈值：90%
结束方式：已达到输出条件 / 用户主动停止追问
关键阻塞项：0 / n
非阻塞待确认项：n
适用领域：……
```

随后给出一句状态说明：

- 自动结束：说明达到门槛，不再追问。
- 主动停止：说明未达门槛的影响，并承诺以下内容严格区分事实与假设。

## 3. 完整交付结构

### 3.1 需求摘要

用 3 至 8 句话说明：

- 当前问题。
- 目标用户或服务对象。
- 拟议方案。
- 核心价值。
- 当前范围和关键约束。

### 3.2 一句话需求定义

使用结构：

```text
为【目标用户】在【关键场景】提供【核心能力】，解决【当前问题】，从而实现【可衡量结果】。
```

避免使用“优化体验”“提升效率”等无法独立验收的空泛表述。

### 3.3 背景、现状与问题

包括：

- 当前流程或替代方案。
- 主要痛点及证据。
- 为什么现在做。
- 不做的成本或风险。
- 已知限制。

### 3.4 目标与成功指标

分别列出：

| 类型 | 目标 | 指标 | 当前基线 | 目标值 | 观察周期 |
|---|---|---|---|---|---|
| 用户目标 |  |  |  |  |  |
| 业务目标 |  |  |  |  |  |
| 产品或流程目标 |  |  |  |  |  |
| 护栏目标 |  |  |  |  |  |

没有基线或目标值时明确写“待基线测量”，不要编造数字。

### 3.5 用户、角色与责任

| 角色 | 目标 | 核心任务 | 查看范围 | 可执行动作 | 最终责任 |
|---|---|---|---|---|---|

必要时补充：

- 主要用户和次要用户。
- 角色切换。
- 外部合作方或系统角色。
- rasic 或 raci 责任矩阵。

### 3.6 使用场景、触发与前置条件

对每个核心场景说明：

| 场景 | 触发事件 | 前置条件 | 用户意图 | 预期结果 | 使用频率或紧急度 |
|---|---|---|---|---|---|

同时列出不允许进入流程的条件。

### 3.7 范围与优先级

分为：

- 本期必须：没有这些能力就无法形成闭环。
- 本期应该：重要但可以有临时方案。
- 本期可选：资源允许时完成。
- 明确不做：防止范围反复。
- 后续版本：保留方向但不纳入本期承诺。

可使用 must / should / could / won’t，但用中文解释。

### 3.8 前置依赖与约束

| 依赖或约束 | 类型 | 负责人 | 当前状态 | 失败影响 | 兜底方案 |
|---|---|---|---|---|---|

覆盖适用项：系统、数据、人员、供应商、政策、预算、场地、库存和时间。

### 3.9 信息架构、页面或触点清单

数字产品使用：

| 页面或组件 | 使用角色 | 入口 | 核心任务 | 出口或下一步 |
|---|---|---|---|---|

非数字流程使用：

| 触点或节点 | 参与者 | 渠道或地点 | 输入 | 动作 | 输出 |
|---|---|---|---|---|---|

### 3.10 功能模块或能力清单

| 编号 | 模块或能力 | 需求说明 | 使用角色 | 优先级 | 依赖 |
|---|---|---|---|---|---|

需求说明聚焦用户结果，不只写组件名称。

### 3.11 主流程

先给出一条端到端动词链，再给出步骤表：

```text
触发 → 进入 → 输入 → 校验 → 提交 → 处理 → 结果 → 通知 → 归档
```

| 步骤 | 责任角色 | 用户或系统动作 | 输入 | 规则 | 输出或状态 |
|---|---|---|---|---|---|

主流程必须有明确起点、终点和结果责任人。

### 3.12 分支、失败与补救流程

至少覆盖适用项：

- 驳回。
- 退回修改。
- 撤回或取消。
- 超时或升级。
- 重试。
- 部分成功。
- 重复提交。
- 并发修改。
- 数据失效。
- 外部依赖失败。
- 人工干预与补偿。

使用格式：

| 触发条件 | 当前状态 | 系统行为 | 用户可见反馈 | 恢复或终态 | 责任人 |
|---|---|---|---|---|---|

### 3.13 业务规则

每条规则使用唯一编号：

```text
r-01 规则名称
条件：……
处理：……
例外：……
配置方：……
生效范围：……
验收关联：ac-xx
```

复杂规则附决策表。明确边界是否包含阈值本身，例如“大于 3 天”和“3 天及以上”不能混用。

### 3.14 状态模型

| 状态 | 进入条件 | 页面或触点表现 | 允许操作 | 禁止操作 | 下一状态 | 超时或恢复 |
|---|---|---|---|---|---|---|

再用文字状态图表达：

```text
草稿 → 已提交 → 处理中 → 已完成
              ├→ 已驳回
              ├→ 已取消
              └→ 处理失败 → 待重试 / 人工处理
```

检查不可达状态、死循环和没有出口的中间状态。

### 3.15 详细交互逻辑

按页面、触点或节点逐一描述：

```text
页面或触点：名称
目标：用户在这里完成什么
入口：从哪里进入
默认状态：首次进入看见什么
信息结构：从上到下或从左到右的内容
主要操作：点击、输入、选择、扫描、提交或交接后的行为
启用条件：按钮或动作何时可用
校验：校验时机、规则和定位方式
成功反馈：提示、状态变化、跳转和数据刷新
失败反馈：错误信息、重试、保留输入和人工帮助
退出与返回：未保存内容、返回位置和状态保留
权限差异：不同角色看见或能做什么
埋点：关键动作对应的事件
```

对高风险动作说明二次确认、可撤销性和日志。

### 3.16 字段、数据、内容或物料设计

数字信息使用：

| 字段 | 类型 | 必填 | 默认值 | 来源 | 校验 | 展示或编辑权限 | 保存与脱敏 |
|---|---|---:|---|---|---|---|---|

线下或运营物料使用：

| 物料或凭证 | 数量或规格 | 来源 | 使用节点 | 保管人 | 核销或回收 | 异常处理 |
|---|---|---|---|---|---|---|

指标数据另列口径、粒度、刷新和负责人。

### 3.17 权限与数据范围矩阵

| 角色 | 查看 | 新增 | 编辑 | 删除 | 审批或执行 | 导出 | 配置 | 强制干预 |
|---|---:|---:|---:|---:|---:|---:|---:|---:|

同时说明：

- 本人、团队、部门、组织、租户或全局数据范围。
- 临时授权和代理。
- 离职或角色变化后的权限回收。
- 敏感数据和高风险操作控制。

### 3.18 通知、提醒与消息

| 事件 | 接收者 | 渠道 | 触发时机 | 内容要点 | 去重或频控 | 点击后去向 |
|---|---|---|---|---|---|---|

说明失败重发、免打扰、升级和已读状态中适用的部分。

### 3.19 操作日志、审计与可追溯性

说明记录：

- 操作人和代理身份。
- 时间、来源、设备或请求标识。
- 操作前后值。
- 原因、附件或审批意见。
- 规则和流程版本。
- 查询权限和保留期限。

### 3.20 接口、系统集成与技术能力

按业务目标而非随意技术拆分列出：

| 接口或集成 | 调用方向 | 目的 | 关键输入 | 关键输出 | 同步方式 | 错误与补偿 | 安全要求 |
|---|---|---|---|---|---|---|---|

对重要接口补充：

- 认证与授权。
- 幂等。
- 超时与重试。
- 错误码和可重试性。
- 版本兼容。
- 监控与追踪。

没有足够技术信息时写“接口能力需求”，不要擅自指定框架、数据库或云服务。

### 3.21 非功能需求

只列适用且可验证的要求：

| 类别 | 要求 | 验证方式 |
|---|---|---|
| 性能 |  |  |
| 容量与并发 |  |  |
| 可用性与容灾 |  |  |
| 安全与隐私 |  |  |
| 兼容性 |  |  |
| 可访问性 |  |  |
| 国际化与时区 |  |  |
| 数据保留与删除 |  |  |
| 监控与告警 |  |  |

目标数值未确认时标记待测量或待技术评估，不编造指标。

### 3.22 埋点、指标与运营观察

| 事件或指标 | 定义 | 触发时机或公式 | 关键属性 | 使用者 | 决策用途 |
|---|---|---|---|---|---|

区分：

- 结果指标。
- 过程漏斗。
- 质量指标。
- 风险护栏。
- 系统健康指标。

对数据口径注明时区、去重对象、时间窗口和版本。

### 3.23 验收标准

核心场景使用 given / when / then：

```gherkin
ac-01 场景名称
Given [已知条件]
And [其他前置条件]
When [用户或系统动作]
Then [可观察结果]
And [状态、数据、通知或日志结果]
```

验收至少覆盖：

- 主流程成功。
- 一个关键规则边界。
- 一个权限限制。
- 一个失败或恢复场景。
- 一个重复或并发场景（适用时）。
- 一个通知、日志或数据结果（适用时）。

### 3.24 文字版低保真原型

按 [text-wireframe-notation.md](text-wireframe-notation.md) 输出：

- 页面或触点结构图。
- 主流程或角色泳道。
- 关键状态变体。
- 每个原型的交互说明。

只绘制支持核心闭环和高风险分支的关键页面或节点，不把所有次要页面都画成重复框架。

### 3.25 决策、假设、待确认与风险

使用四张清单：

#### 已确认关键决策

| 决策 | 决策内容 | 影响范围 |
|---|---|---|

#### 已接受或暂定假设

| 假设 | 状态 | 采用原因 | 若不成立的影响 |
|---|---|---|---|

#### 非阻塞待确认项

| 事项 | 建议确认阶段 | 影响 |
|---|---|---|

#### 风险与缓解

| 风险 | 概率或不确定性 | 影响 | 预防 | 发生后处理 | 责任人 |
|---|---|---|---|---|---|

主动停止时另列“关键阻塞项”，放在此章节最前面。

### 3.26 发布、迁移与运营准备

适用时包括：

- 历史数据迁移。
- 灰度范围和开关。
- 培训、帮助文档和客服口径。
- 监控和上线值守。
- 回滚条件和回滚方式。
- 旧流程停用和过渡期。
- 规则或配置初始化。

### 3.27 需求追踪矩阵

复杂需求附简化追踪表：

| 目标 | 功能或流程 | 规则 | 指标 | 验收 |
|---|---|---|---|---|

用它检查是否存在：没有目标支撑的功能、没有功能支撑的目标、没有验收的规则或没有数据来源的指标。

## 4. 主动停止时的调整

用户主动停止追问后：

1. 不再提出任何问题，包括最终结尾。
2. 在文档开头列出所有关键阻塞项及影响。
3. 对无法省略的设计缺口使用最保守、最可逆的暂定假设。
4. 给每个暂定假设写出“不成立时需修改的部分”。
5. 验收标准只覆盖已有信息能够支持的范围。
6. 原型可以展示建议方案，但在标题中标记“基于暂定假设”。
7. 不得使用“已完成”“已确认”描述未确认内容。

## 5. 输出前一致性检查

最终输出前静默核对：

- 一句话需求与目标一致。
- 范围中的每项能力在功能或流程中有落点。
- 主流程中的每个动作都有角色、数据和状态支持。
- 规则编号与状态、交互和验收一致。
- 页面或触点清单覆盖主流程中的关键动作。
- 权限矩阵允许必要动作并阻止越权动作。
- 接口或依赖能够支持交互承诺。
- 通知和日志覆盖高风险节点。
- 指标能够反映目标，而不是只统计点击。
- 原型与功能、规则和状态一致。
- 所有假设均显式标记。
- 没有把非本期内容混入本期验收。
