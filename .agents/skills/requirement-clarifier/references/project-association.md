# 项目关联与需求类型判定

本文件用于需求可能涉及已有系统时，确定是否读取项目、如何建立分析基线，以及如何区分迭代需求与全新需求。

## 1. 先确定用户选择

项目关联只有两种工作方式：

| 方式 | 可使用的信息 | 必须输出 | 禁止行为 |
|---|---|---|---|
| 关联项目分析 | 当前对话材料，以及项目内与需求相关的文档、页面、代码、配置和测试 | 分析基线、项目上下文卡、证据位置、AS-IS/TO-BE/GAP | 未经授权修改业务代码；把代码实现当作目标规则 |
| 不关联项目分析 | 用户主动提供的描述、文档、截图和规则 | 信息来源、未核验边界；如为迭代需求仍输出 AS-IS/TO-BE/GAP | 扫描项目；虚构项目事实；自动判定为全新需求 |

只有以下情况可视为用户已选择：

- “关联当前项目分析”“结合仓库/代码看看”“基于这个项目整理”或同等明确表达：关联项目分析。
- “不要读取项目”“只按需求描述/截图整理”“不关联项目”或同等明确表达：不关联项目分析。

仅提到“现有系统”“迭代”“已有页面”时，仍需给出一次两选一；不得替用户选择。用户中途切换关联方式时，保留已确认的业务结论，重新检查 AS-IS、GAP、影响范围和证据边界。

## 2. 关联项目分析流程

### 2.1 锁定分析基线

按可获得程度记录：

```text
关联方式：关联项目分析
项目/仓库：
分析范围：
分支/版本：
Commit：
环境/时间点：
无法确认项：
```

分析基线用于说明“以哪个版本的系统作为 AS-IS”。如果用户无法提供版本号，可使用“用户指定的当前生产环境”或“本次会话读取到的当前工作区”，但必须按事实表述，不能编造版本或 Commit。

### 2.2 建立项目上下文卡

只检查与需求有关的内容：

| 维度 | 检查内容 | 典型证据 |
|---|---|---|
| 业务入口 | 菜单、路由、页面入口 | 路由与页面文件 |
| 角色权限 | 角色、操作权限、数据范围 | 权限配置与服务端校验 |
| 主流程 | 入口到结果的现有步骤 | 页面交互、服务调用、测试 |
| 状态 | 当前状态及转换条件 | 枚举、状态机、业务逻辑 |
| 数据对象 | 核心对象、字段和关系 | 类型、模型、接口定义 |
| 服务能力 | 接口、校验、错误和部分成功 | API 与测试 |
| 历史约束 | 兼容、迁移、旧入口和用户习惯 | 发布文档、旧逻辑、数据脚本 |

区分【项目事实】【合理推断】【待确认】。项目事实也只能作为 AS-IS 证据，不自动升级为 TO-BE。

## 3. 不关联项目分析流程

记录：

```text
关联方式：不关联项目分析
分析依据：用户描述/用户提供的 PRD/截图/流程图
项目核验：未执行
证据边界：页面、接口、权限、状态和历史数据以用户材料为准
```

如果用户描述的是已有系统改造，继续整理：

- AS-IS：根据用户描述或材料提取，并标注未与项目核验。
- TO-BE：根据本次已确认目标设计。
- GAP：当前可判断的保留、新增、修改、废弃。
- 变更影响：只写可判断项；项目级影响标记待核验。

不关联项目可能降低 AS-IS 和影响分析的确定性，但不阻止需求目标、角色、流程、规则、页面和验收的整理。只有未核验信息本身符合 P0 严格定义时，才把对应业务问题标为 P0；不能把“不关联项目”这个选择标为 P0。

## 4. 需求类型判定

按业务关系判断，而不是按页面数量或代码位置判断：

| 判定信号 | 存量改造型迭代 | 存量系统新增能力 | 全新需求 |
|---|---|---|---|
| 已有业务对象 | 修改其行为 | 读取或扩展使用 | 不依赖 |
| 已有主流程/状态 | 直接改变 | 新增旁路或新阶段 | 独立建立 |
| 已有角色/权限 | 修改或复用 | 依赖 | 基本独立 |
| 历史数据/兼容 | 通常需要 | 可能需要 | 通常不需要 |
| 技术/视觉复用 | 可有可无 | 可有可无 | 可以复用，但不足以构成迭代 |

满足以下任一项时，默认按迭代需求检查影响：

- 修改已有页面、入口、流程、规则或结果。
- 操作已有业务对象或改变其状态。
- 依赖已有角色、权限、数据范围、接口或统计口径。
- 需要处理历史数据、旧入口、新旧兼容、灰度或回滚。

只有业务目标、角色、对象、流程和状态基本独立，且不需要历史兼容时，才判定为全新需求。证据不足时标记【待确认】或采用显式合理假设，不因为用户选择不关联项目而直接判定。

## 5. 正式输出要求

在正式方案的“需求理解”中记录：

- 项目关联方式。
- 需求类型及判定依据。
- 分析基线（关联项目时）。
- 分析依据与证据边界。
- 项目上下文卡或未核验说明。
- AS-IS、TO-BE、GAP（迭代需求）。

关联方式只决定证据来源，不能替代目标、角色、流程、权限、状态和验收检查。
