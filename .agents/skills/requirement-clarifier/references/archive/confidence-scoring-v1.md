## 1. 评分目的

使用两个分数避免把“信息很多”误认为“方案已经稳定”：

- 信息完整度：适用维度是否被覆盖。
- 方案把握度：已有信息是否明确、一致、可执行并足以支撑设计。

分数是基于需求证据的启发式估计，不是统计概率。只在获取新信息、确认决定、解决冲突或发现新风险时变化。允许分数下降。

## 2. 维度权重

| 维度 | 默认权重 |
|---|---:|
| 背景、问题、目标与价值 | 10 |
| 用户、角色与利益相关者 | 8 |
| 场景、触发与前置条件 | 8 |
| 范围、优先级与版本边界 | 10 |
| 主流程与关键分支 | 14 |
| 业务规则与决策逻辑 | 12 |
| 数据、字段、内容与物料 | 8 |
| 权限、责任、安全与合规 | 8 |
| 状态、异常、边界与恢复 | 8 |
| 页面、触点与交互方式 | 6 |
| 依赖、集成与非功能要求 | 4 |
| 指标、运营与验收 | 4 |
| **合计** | **100** |

根据需求类型微调权重时遵守：

- 单个维度最多上下调整 4 分。
- 主流程权重不得低于 10。
- 目标、用户、范围和验收的合计不得低于 28。
- 涉及资金、安全、隐私或监管时，权限与合规权重至少为 10。
- ai 自动执行高风险动作时，权限、安全、人工确认与评测的合计权重至少为 18。

某维度确实不适用时，将其标记为“不适用”，并按比例把权重归一化到其余适用维度。不得为了抬高分数随意标记不适用。

## 3. 信息完整度

### 3.1 覆盖系数

为每个适用维度选择一个覆盖系数：

| 状态 | 系数 | 判断标准 |
|---|---:|---|
| 完整覆盖 | 1.00 | 该维度的核心问题和关键边界都有答案 |
| 大部分覆盖 | 0.80 | 主体明确，只剩低影响细节 |
| 部分覆盖 | 0.50 | 有方向，但仍缺少重要规则或分支 |
| 仅有线索 | 0.20 | 只有一句描述、推测或间接信息 |
| 未覆盖 | 0.00 | 没有可用信息 |

计算：

```text
信息完整度 = Σ（归一化权重 × 覆盖系数）
```

四舍五入为整数百分比。

### 3.2 覆盖不等于确认

以下内容可以增加完整度，但不应等量增加把握度：

- 助手提出但用户未确认的合理假设。
- 文档中存在但与用户口头表述可能冲突的规则。
- 已知有某个外部依赖，但尚未验证其能力。
- 已列出异常场景，但处理方式尚未决定。

## 4. 方案把握度

对每个适用维度同时评估“证据确定性”和“可执行性”。

### 4.1 证据确定性

| 证据状态 | 系数 |
|---|---:|
| 用户明确确认，且与其他信息一致 | 1.00 |
| 用户明确接受推荐默认方案 | 0.85 |
| 权威文档明确，但尚未由用户确认适用性 | 0.75 |
| 基于上下文的稳定推断 | 0.60 |
| 助手建议或弱推断，未确认 | 0.30 |
| 未知 | 0.00 |
| 存在直接冲突 | 0.20，且检查是否构成阻塞 |

### 4.2 可执行性

| 可执行状态 | 系数 |
|---|---:|
| 条件、责任、输入、结果和验收均明确 | 1.00 |
| 方案基本可执行，只缺低影响细节 | 0.85 |
| 只有高层方向，仍需重要设计决定 | 0.60 |
| 无法据此设计、开发、测试或执行 | 0.30 |

### 4.3 计算

```text
维度把握度 = 覆盖系数 × 证据确定性 × 可执行性
方案把握度 = Σ（归一化权重 × 维度把握度）
```

对明显冲突、未识别依赖或不可测试的结论进行额外下调，但不要重复扣分。最终四舍五入为整数百分比。

### 4.4 已确认维度

一个维度同时满足以下条件时计为“已确认”：

- 覆盖系数至少 0.80。
- 证据确定性至少 0.85。
- 不存在会改变该维度结论的冲突。

进度区块显示“已确认维度：x/适用维度数”。

## 5. 关键阻塞项与上限

关键阻塞项会限制方案把握度，防止分数被大量低价值细节抬高。

| 情况 | 方案把握度上限 |
|---|---:|
| 核心问题、主要用户或主流程任一完全未知 | 69% |
| 存在两个及以上关键阻塞项 | 79% |
| 存在一个关键阻塞项 | 89% |
| 无关键阻塞，但仍有重要未确认项 | 94% |
| 只有低影响待确认项 | 98% |

涉及资金、安全、隐私、医疗、法律、未成年人或不可逆自动执行时，如果必要控制措施未确认，至少标记一个关键阻塞项。

用户主动输入“停止追问，按当前信息输出”时，只绕过自动结束门槛，不绕过评分规则。继续显示真实分数和阻塞上限，然后基于现有信息输出。

## 6. 进度展示

每次回复都显示：

```text
【需求澄清进度】
信息完整度：xx%
方案把握度：xx%
目标阈值：90%
当前阶段：……
已确认维度：x/12
关键阻塞：……
本轮新增确认：……
剩余重点：……
```

### 阶段名称参考

- 初始解析：刚接收需求，正在识别对象和目标。
- 目标与范围：确认问题、用户、成功标准和版本边界。
- 核心流程：建立端到端主流程和责任链。
- 规则与数据：明确决策规则、字段、来源、权限和状态。
- 异常与交互：处理边界、恢复、页面或触点行为。
- 验收与收口：补齐指标、依赖、验收和一致性。
- 需求输出：达到门槛或用户主动停止，正在交付方案。

阶段由当前工作内容决定，不仅由分数决定。

### 展示变化依据

当分数发生明显变化时，在“本轮新增确认”中说明原因，例如：

- 已明确审批层级，主流程和权限两个维度同步提高。
- 新发现支付退款依赖尚未验证，方案把握度从 86% 降至 80%。
- 用户修改了日期规则，原有状态和验收场景需要重算。

不要展示冗长的内部计算表，除非用户明确要求查看评分明细。

## 7. 自动结束门槛

默认目标阈值为 90%。只有全部满足才自动结束：

1. 信息完整度大于等于目标阈值。
2. 方案把握度大于等于目标阈值。
3. 关键阻塞项为零。
4. 主流程及关键失败分支闭环。
5. 关键角色、规则、数据、权限和状态一致。
6. 至少有一个业务成功指标或结果指标。
7. 至少有一组可测试验收场景。

不要为了追求 100% 追问微小细节。达到门槛且剩余事项不会改变整体方案时，停止追问并将其列为“非阻塞待确认项”。

除非所有适用维度均由用户确认、无假设、关键依赖已验证且验收证据充分，否则不要显示 100%。通常成熟方案显示 90% 至 98%。

## 8. 评分校准示例

### 示例 A：一句需求

输入：

> 做一个供应商审批功能。

判断：

- 有功能方向，但目标、角色、流程、规则和验收未知。
- 信息完整度约 10% 至 18%。
- 方案把握度约 5% 至 12%。
- 当前阻塞：目标和审批责任未知。

### 示例 B：流程基本明确，但规则冲突

已有信息：角色、入口、主流程、字段和页面均明确；文档写“金额超过 1 万由财务审批”，用户又说“超过 5 万才需要财务”。

判断：

- 信息完整度可以达到 80% 以上。
- 方案把握度应明显更低，因为关键规则冲突。
- 该冲突影响审批链、权限、状态、通知和验收，属于关键阻塞项，方案把握度不得超过 89%。

### 示例 C：达到输出条件

已有信息：目标、用户、范围、主流程、规则、权限、数据、异常、交互、依赖和验收均明确；只剩消息模板文案和导出文件名未定。

判断：

- 信息完整度约 93% 至 97%。
- 方案把握度约 90% 至 95%。
- 剩余事项是非阻塞细节，应停止追问并输出。

### 示例 D：用户提前停止

当前完整度 62%，把握度 55%，有两个关键阻塞项。用户要求停止。

处理：

- 进入需求输出阶段。
- 保持 62% 和 55%，说明未达 90%。
- 把阻塞项放在最终方案开头。
- 对必须填补的部分采用“暂定假设”，标注影响和后续确认建议。
- 不再提出任何问题。
