# V2 提问参考

本文件用于选择少量、高价值问题，不要求把九步分析框架执行成九轮对话。已知内容直接提取，只追问会改变方案或阻塞闭环的内容。

## 项目关联方式选择

需求可能涉及已有系统时，先检查用户是否已经明确选择关联项目分析或不关联项目分析。已选择时不得重复询问；未选择时只问一个两选一问题：

```text
本次希望采用哪种分析方式？
A. 关联项目分析（已有系统迭代推荐）—只读核验项目现状，并给出分析基线和证据位置。
B. 不关联项目分析—只按你提供的描述或材料整理，项目现状标记为未核验。
```

这是一项分析方式选择，不是业务 P0。选择不关联项目不能自动推导为全新需求，也不能因为证据较少而停止需求整理。只有具体缺失的角色、权限、状态、统计口径或验收信息符合 P0 严格标准时，才标记对应问题为 P0。

## P0 判断标准

不确认会导致根本性方案差异，且符合以下至少一项：

- 真实使用者无法确定。
- 用户最终结果无法确定。
- 会分成两套以上完全不同的主流程。
- 存在越权、数据泄露或严重权限风险。
- 核心状态不能进入、退出或闭环。
- 核心统计口径差异会使结果失真。
- 无法判断是否完成或验收通过。
- 页面承载选择会连带改变核心结果、关键权限、状态闭环或用户已授权的实施范围。

判定前问自己：“如果采用一个保守、可逆假设，主流程是否仍能稳定设计？”如果可以，通常不是 P0。

只是在新增页面、现有页面、弹窗或抽屉之间选择，且不改变核心结果、权限和状态闭环时，通常是 P1 或 P2，不得仅因页面形态不同升级为 P0。

## P1 问题示例

- 是否允许撤回或再次修改？
- 是否需要历史记录、导出或批量处理？
- 通知接收人和触发时机是什么？
- 默认排序、常用筛选和列展示顺序是什么？
- 是否需要保留修改前后值？

这些可使用显式合理假设推进，并记录假设不成立的影响。

## P2 问题示例

- 弹窗还是抽屉。
- 图表颜色、圆角、密度和卡片样式。
- 默认展开或收起。
- 非核心筛选项和次要列宽。
- 普通成功提示的文案风格。

P2 不应阻塞需求分析或原型说明。

## 不应该追问的问题

- 用户已明确回答的内容。
- 可从正式文档、页面、截图或代码中直接提取的 AS-IS 信息。
- 尚未进入页面设计却询问视觉风格。
- 与核心闭环无关的接口、字段、指标或角色。
- 仅为了让每一步都有用户回复而提出的形式化问题。
- 可在正式方案中补充的完整 Given/When/Then 文案。

## 不同需求类型的关键问题

### 简单页面优化

- 现有问题是找不到、看不懂，还是无法完成操作？
- 主要使用者和成功结果是什么？
- 是复用当前页面还是必须新增入口？

### 已有系统改造

- AS-IS 实际如何运行，哪些逻辑需保留？
- TO-BE 与现状的核心差异是什么？
- GAP 是保留、新增、修改还是废弃？
- 历史数据、旧入口和原有习惯如何处理？

### 多角色审批

- 谁提交、谁审批、谁最终负责？
- 每个角色的数据范围和禁止操作是什么？
- 驳回、退回、撤回、超时和代理审批如何改变状态？
- 日志、审计和追责信息需记录什么？

### 数据看板

- 指标用来做什么决策？
- 分子、分母、去重对象、时间窗口和时区是什么？
- 数据来源、更新频率、延迟和历史回算规则是什么？
- 哪些角色可以看到哪个粒度？

### AI 功能

- AI 结果用于建议还是自动决策？
- 错误结果的业务损失和人工兜底是什么？
- 输入、输出和日志中是否包含敏感信息？
- 响应时间、生成成本、内容安全和可解释性如何验收？

## 特殊回复的处理

### 用户回复“不知道”

1. 不立即停止。
2. 给出 2—4 个互斥或差异清晰的选项。
3. 说明每个选项会改变的流程、权限或页面。
4. 推荐风险最低、最可逆的默认值。
5. 非 P0 可记为【合理假设】继续；P0 则说明阻塞影响。

### 用户只回复“确认，下一步”

- 将上一轮展示的推荐结论记为已确认。
- 自动检查后续步骤，不重复展示全部历史。
- 没有 P0 时可连续跨过多个已满足步骤。
- 达到停止条件后直接输出，不询问“是否现在输出”。

### 用户要求停止追问

立即停止，不再以任何形式追问。输出未解决 P0、合理假设、假设不成立的影响和无法承诺内容，然后按当前信息生成可用方案。

## 避免重复提问

每次提问前检查：

1. 当前对话是否已回答。
2. 用户材料是否已提供。
3. 问题是否真正会改变方案。
4. 是否可以作为合理假设。
5. 是否可与同一决策下的其他问题合并。
6. 用户是否已选择项目关联方式，避免再次询问。

## 用选项降低回答成本

选项应互斥、差异清晰，最多 4 个，推荐项置前。格式：

```text
1.【P0｜主流程结果】审批通过后由谁执行最终生效？
A. 系统自动生效（推荐）—链路短，需保证前置校验完整
B. 运营人工生效—风险可控，但增加待办和时延
C. 申请人二次确认—用户可控，但主流程更长
影响：决定终态、责任人和页面出口。
```
